# Проект «Мой день»

Перед началом любой задачи сначала полностью прочитай этот файл.

## Главная цель

Создать красивое, спокойное и удобное приложение, которое человек открывает каждый день, чтобы понять, что его ждёт сегодня.

## Принципы продукта

- Простота важнее количества функций.
- Главный экран показывает только сегодняшний день.
- Календарь — основной источник событий и задач.
- Дополнительные модули пользователь сможет включать и выключать.
- Новая функция должна приносить реальную пользу.
- Приложение должно быть внимательным, а не перегруженным.

## Работа с проектом

- Один этап = одна задача.
- Не менять файлы, не относящиеся к текущему этапу.
- Не создавать резервные копии и дубли файлов без явной просьбы.
- Не удалять файлы без подтверждения пользователя.
- Если решение можно сделать проще — предложить более простой вариант.
- Если пользователь предлагает слабое или рискованное решение — объяснить почему и предложить лучшее.
- Не соглашаться автоматически.

## Рабочий процесс

При начале каждой новой задачи в новом агенте:

1. Сначала очистить встроенный терминал Cursor командой `cls`.
2. Затем изучить задачу.
3. Показать только план.
4. Код не менять до команды «Можно выполнять».

После завершения задачи:

1. Пользователь проверяет результат.
2. Нажимает Keep All.
3. Запускает `finish-module.bat`.
4. После нажатия любой клавиши (`pause`) терминал автоматически очищается (`cls`) — это штатное поведение скрипта, а не ошибка.

После успешного завершения модуля (проверка → Keep All → `finish-module.bat`) ответ обязательно завершать **только** этим блоком:

```
════════════════════════════

✅ МОДУЛЬ ЗАВЕРШЁН

✔ Проверено пользователем
✔ Keep All выполнен
✔ GitHub синхронизирован

➡ Следующий шаг:
Создайте новый чат Cursor.

════════════════════════════
```

После этого:

- не писать «Дальше: проверка → Keep All → finish-module.bat»;
- не давать никаких других рекомендаций по текущему модулю;
- не предлагать доработки текущего модуля.

Считать обсуждение текущего модуля завершённым.

Новый чат = новый модуль.

## После каждого этапа

1. Проверить код.
2. Исправить найденные ошибки.
3. Повторить проверку.
4. Перечислить изменённые файлы.
5. Написать, что удалось проверить автоматически.
6. Написать, что не удалось проверить автоматически.
7. Дать короткий чек-лист ручной проверки.
8. После успешных проверок запустить локальный сервер.
9. Открыть index.html через адрес локального сервера в браузере.
10. Не открывать проект через file://, если доступен локальный сервер.
11. Если автоматическое открытие браузера невозможно, написать точный адрес для ручного открытия.
12. Не завершать локальный сервер до ручной проверки пользователя, если это возможно.

## Git

Никогда не выполнять:

- git add
- git commit
- git push

Git выполняется только пользователем вручную.

# Контроль риска перед изменениями

Перед выполнением любой задачи Cursor обязан сначала оценить её влияние на проект.

## 1. Сначала анализ, потом код

До изменения файлов Cursor должен сообщить:

ПЛАН ИЗМЕНЕНИЯ

- цель задачи;
- какие файлы потребуется изменить;
- какие уже работающие модули могут быть затронуты;
- есть ли риск потерять существующую функциональность;
- есть ли более безопасный способ выполнить задачу;
- нужна ли сначала резервная точка Git.

После этого Cursor должен остановиться и ждать подтверждения пользователя:

«Можно выполнять».

Без подтверждения код не менять.

## 2. Обязательное предупреждение

Cursor обязан предупредить пользователя до выполнения, если:

- задача затрагивает более трёх файлов;
- меняется архитектура;
- меняются пути к файлам;
- меняется формат данных;
- меняется существующее хранилище;
- объединяются разные источники данных;
- удаляется или заменяется работающая логика;
- новая задача может повлиять на утверждённый модуль;
- одновременно запрошено несколько разных изменений;
- есть риск, что потребуется откат.

Формат предупреждения:

⚠ РИСК ИЗМЕНЕНИЯ

Что может быть затронуто:
- ...

Почему это риск:
- ...

Безопасный вариант:
- ...

Жду подтверждения. Файлы пока не меняю.

## 3. Одна задача за один раз

Если в запросе несколько изменений, Cursor не должен выполнять их одновременно.

Он обязан:

1. разделить запрос на отдельные этапы;
2. предложить порядок;
3. выполнить только первый этап после подтверждения;
4. остановиться для проверки пользователя.

## 4. Сохранение работающего результата

Если текущая версия работает, а последнего Git-коммита после проверки ещё нет, Cursor должен сначала написать:

«Текущий рабочий результат ещё не зафиксирован в Git. Рекомендую сначала выполнить commit и push».

Cursor не должен выполнять Git самостоятельно без прямой команды пользователя.

## 5. Проверка перед завершением

После изменения Cursor обязан проверить не только новую функцию, но и связанные утверждённые функции.

Для календаря минимальная регрессионная проверка:

- личный день рождения;
- поздравления;
- важная дата;
- официальный праздник;
- памятная дата;
- переходящая церковная дата;
- пустой месяц;
- переход в экран дня;
- возврат в исходный месяц.

Если проверить что-либо невозможно, Cursor должен честно написать это в отчёте.

## 6. Запрет на самостоятельные улучшения

Не выполнять:

- рефакторинг «заодно»;
- архитектурную уборку вместе с новой функцией;
- изменение дизайна вместе с логикой;
- перенос файлов во время исправления ошибки;
- замену работающего решения без прямой необходимости.

## 7. Режим экономии токенов

Ответы должны быть краткими.

До подтверждения показывать только:

=== ПЛАН ПЕРЕД ИЗМЕНЕНИЕМ ===

Файлы

Риски

Безопасный план

Жду команды: «Можно выполнять».

После выполнения показывать только:

=== ИТОГ ПОСЛЕДНЕЙ ЗАДАЧИ ===

Изменённые файлы

Что сделано

Что проверено

Что намеренно не менялось

## 8. При сомнении — остановиться

Если Cursor не уверен в структуре данных, источнике ошибки или последствиях изменения, он не должен угадывать и редактировать код.

Он должен остановиться, кратко описать неопределённость и предложить безопасную проверку без изменения файлов.
